Plus d’une heure… (STOPPER)

PROBLEME identifié : ce n’est pas l’algorithme qui est super lent, car les itérations s’enchainent assez vite (en tout cas relativement à la taille du graphe), mais c’est la faute de networkx : NetworkX est monocœur et extrêmement gourmand en RAM (environ 5 à 10 fois le poids du fichier brut). Pour 5 millions d'arêtes, il sature probablement déjà le driver Colab avant même que Spark ne commence à travailler.
Actuellement, on fais : Fichier TXT -> NetworkX -> RDD. 
Solution : retirer complètement Networkx, on laisse Spark gérer le flux de données !

V2 : même chose sans utiliser Networkx
Pourquoi c'est mieux :
1.	Mémoire : On ne stocke plus d'objet nx.DiGraph. On ne manipule que des RDD, que Spark peut décharger sur le disque si la RAM manque.
2.	Vitesse : sc.textFile est parallélisé nativement.
3.	Stabilité : J'ai ajouté .cache() après chaque passe dans forward_pass. C'est l'étape cruciale : sans cela, à l'itération 10, Spark recalcule 10 fois le début de l'algorithme.
Note : utilisation de unpersist() pour libérer la RAM à chaque fin d’étape

Pourquoi le Secondary Sort est-il plus lent ici ?
Il y a deux raisons techniques probables :
1.	Le coût du sorted(list(values)) : Dans le dataset web-Google, la plupart des nœuds ont peu de voisins, mais quelques "hubs" en ont énormément. Convertir un itérateur Spark en liste Python pour la trier consomme beaucoup de CPU et de RAM. Si la liste est déjà presque triée ou très courte, l'overhead du tri est plus lourd que le gain.
2.	La sérialisation Python/Java : Le tri se passe côté Python. Faire transiter de gros volumes de données entre la JVM (Spark) et Python pour les trier ralentit l'exécution
•  L'overhead de la liste : values est un itérateur. En faisant list(values), tu forces Spark à charger tous les voisins d'un nœud en mémoire Python d'un coup.
•  Le coût du tri : Python utilise Timsort ($O(n \log n)$). Si un nœud est un énorme "hub" (ce qui arrive dans les graphes du web), trier 100 000 voisins en Python est plus lent que de simplement les parcourir deux fois comme dans ta version "Sans SS".
